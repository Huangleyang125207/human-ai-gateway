晨会产品投资 石璟岩分享黑森投资

一家叫黑森的主管多头私募，2014 年北京海淀注册，自主发行九号 / 六号 / 三号三只产品（策略复制）。代表产品黑森九号成立以来：

  • 年化 20.52%
  • 最大回撤 13.23%
  • 夏普 1.3073
  • 分红再投资净值 8.856（截至 2026-04-30）

选股策略：基本面多因子检查清单（8 项 · 价值因子权重 70%）

  1. 行业：周期 / 供需 / 成长三视角
  2. 业务：现金流、市场空间、供需格局、渗透率、产品升级、竞争格局、客户粘性
  3. 财务：ROIC & g / 毛利 / 费用 / 净利，同行横比
  4. 竞争：四类护城河（规模、网络、知识产权、转换成本）+ 波特五力
  5. 历史：过去几年增长 + 业绩波动原因 + 先行指标
  6. 增长：新要素（新产品 / 渠道 / 供给 / 需求 / 管理 / 净利率提升）
  7. 估值：3–5 年后值多少？安全边际 + 市场先生 + 不熟不做
  8. 风险：股价操纵、ESG、下跌原因（估值 / 业绩 / 逻辑）

意义：和 yanpai investment-dashboard 自己的因子打分思路高度同源 —— 可作为 dashboard 因子体系的对照样本。"价值因子 70% 权重"这条尤其要记，dashboard 当前因子加权还没明确这个层级。

yanpai协作 启动页样例改东财人气榜 + 后端推送架构

/analyze 之前硬编码三只样例（茅台 / 长江电力 / 宁德）。先想用本地 analyses 表 top-N，被否 —— 我们样本量太小，选股偏单一。改东财源，踩第一个坑：盘前（<9:30）用 push2 clist by 成交额返字段全 "-"，启动页 9 点前就崩。切到 stockrank/getAllCurrentList 实时人气榜，24 h 都有数据。

第二轮架构演化更关键：第一版让 API 层 in-process 5 min 缓存，葱鸭提议"后端统一刷"。改成 celery beat 每 5 分钟拉一次写 Redis，/stocks/hot 直接读 Redis 三层兜底（Redis → memcache → live fetch）。收益是用户响应永远 <50 ms，东财接口压固定不随用户量上升。

pull-based caching → push-based scheduled refresh 是个通用模式，以后任何"轻数据 + 高读频"的接口（行情榜单 / 热搜 / 推荐位）都该走这路子。本来准备直接部署，backfill 在抢 SSH 资源（scp 一直断），决定等 backfill 4 点跑完再统一推。挂了 monitor 持久任务等通知。

12：30 VI

yanpai协作 backfill 把整台机打爆

葱鸭午饭后发现 dash.yanpaidb.cn 打不开，SSH banner exchange timeout —— sshd 自己抢不到 CPU 调度。证据链：早上 10 点 load 21，11 点 backfill 才跑到 6%（sh.600316），12 点 SSH 整个挂掉。2C2G 机器同时扛 8 容器 + sentence-transformers + 长跑 backfill 进程是没头空间的，OOM 把所有进程一起拖入 swap thrashing。腾讯云控制台重启实例救回。

真正的教训是 ETL 不该跟 web 服务塞同一台机。celery 的 refresh_stock_annual 是季频小刷（只 2 年），但 backfill_annual_history 是 11 年全量 ETL，这种活生产 server 不该承担。后续要么独立 ETL 机，要么 backfill 写成分片小任务靠 celery beat 分散触发，而不是一个 7 小时的长进程。

13：00 VII

yanpai协作 本地并发跑 backfill 撞 Clash TUN

葱鸭说"本地跑并发再 scp 上去"。本地 Mac 8 核 16 G，理论 30 分钟搞完云端 7 h 的活，稳赢策略。写了 backfill_annual_parallel.py 用 ProcessPoolExecutor 8 进程并发，smoke test 50 只 4 进程 —— 卡死，bs.login() 25 秒不回。

排错 2 小时：网络通（nc baostock.com 10030 succeeded），import baostock 通，但 login 协议层不回。route get www.baostock.com 显示 gateway 198.18.0.1 / interface utun4 —— Clash TUN 模式接管所有 TCP 出向流量，baostock 用的私有二进制协议被 Clash 当 HTTP 解析失败 → 永久挂起。

这是个通用 Mac 开发陷阱 —— Clash TUN 对纯 socket 协议（baostock / 自研 TCP 服务 / 任何非 HTTP/SOCKS 的二进制协议）都会挂。葱鸭说"我没法修改规则"，试图绕 Clash 不成，2 小时换不来一个 fix，接受沉没成本。以后任何"本地跑"涉及金融数据 API 或私有协议的，先 route get 一下确认不走 utun。

16：30 VIII · 终

yanpai协作 hot-stocks 上线 + 分片 backfill 抢救

放弃本地，回 2C2G 云端。先用 docker cp 把 hot-stocks（东财人气榜 → Redis 推送）和 fetcher 修复 hot-patch 进 worker / api / beat，省掉 web rebuild 的内存峰值 —— /stocks/hot 返大唐发电 / 中际旭创等真热门股，启动页样例不再硬编码茅台 / 长江电力 / 宁德。

backfill 改分片：200 只一片，每片 python 进程独立生命周期，跑完退出释放内存。这样 idle 1.4 GB + 单片 ~300 MB 不会再压崩。挂 monitor 等第一片（~17 min）完成，稳了再循环跑剩下 25 片（~6 小时全量铺底）。

2C2G 跑大 ETL 的姿势 = 进程式分片 + 每片独立内存生命周期，等同人为 OOM-prevention。葱鸭说"现在 2C2G 跑着，要迁移再升 8 G"，这条策略可以撑到迁移。但每次部署或运维都得这么小心翼翼，长期看 8 G 升级值的（每 GB 内存换回 30 min 排查时间）。
