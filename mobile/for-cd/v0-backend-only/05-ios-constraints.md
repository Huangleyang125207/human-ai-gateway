# 05 · iOS 真机约束 —— 视觉设计师必读

> 这一篇不讲产品调性，只讲**真机会咬人的硬约束**。
> Capacitor 把 web 装进 WKWebView，再装进 App Store 审核流。
> 凡是这一篇写"不能"的地方，绕过去 = 设计稿好看但装不进 iPhone，或者上架被拒。
> 凡是写"已经替你处理"的地方，cd 不用操心，照常出视觉。

---

## 1 · 视口 —— 设计的画布到底有多宽

### 真实物理宽度

cd 出稿的画布尺寸**不是凭手感选 375 或 414**，是被这套设备网格锁死的：

| 机型 | 逻辑宽 | 备注 |
|---|---|---|
| iPhone SE / 8 / 8 (二代/三代) | **320** | 在用户群里数量不大，但**不能崩**（横向横滑容器、长 tag 串、tool chip 必须能塞） |
| iPhone 12 / 13 / 14 / 15 / 16（标准款） | **390** | **设计基线，所有 mock 都按 390x844 出** |
| iPhone Pro / Plus / Pro Max | **414 / 428 / 430** | 多出来的横向空间留白即可，不要塞更多内容（不要做"大屏才出现的功能"） |

**iPad 不在本期范围内**。Capacitor 默认 iPhone-only 出包，cd 也不出 iPad 稿。

### Safe area（刘海/灵动岛/Home Indicator）

| 区域 | 占用 | 视觉含义 |
|---|---|---|
| Safe-area-top | **47–59 px** | 顶栏必须从这条线下面开始；状态栏图标（信号/电池）是 iOS 画的，cd 不画，但**顶栏背景色要延伸到屏幕顶**（不留白条），否则视觉断裂 |
| Safe-area-bottom | **34 px** | Home Indicator 那条横线区。底部如果有横置工具条（chat 输入框、tab bar），背景延伸下去但**可点击元素必须在这条线以上** |

设计稿出图的时候建议**画两条参考线**（top safe / bottom safe），让 dev 直观知道"哪儿绝对不能放按钮、但背景必须铺到底"。

> 已经替你处理：CSS `env(safe-area-inset-*)` 这套 dev 会接上，cd 不用算具体像素。你只要在稿子里把 safe-area 的区分标出来，颜色延伸到屏幕边缘的设计意图清楚就行。

### 旋转

**只竖屏**。横屏被 App 配置层锁掉。cd 不用出横屏稿。

## 2 · ATS（App Transport Security）—— 网络链接的隐形过滤器

iOS 默认**拒绝所有 http:// 明文请求**。这条规则会以两种方式渗到设计层：

### a. web_search 出来的链接卡片

AI 通过 web_search tool 召回的结果里很可能带 `http://` URL（老站点、政府站、博客）。

- 后端**自动把 http→https 试着升一次**（mobile-api.js 已实现 `_httpsUpgrade`），用户看到的链接是升级后的。
- 升级失败的链接，会带一个**"链接打不开（站点不支持安全连接）"hint**回到 LLM 那一侧由它解释。
- **cd 这边的约束**：链接卡片的视觉要为"可能打不开"这种降级状态留位置。不要把链接做成铁定能跳的高确定性按钮（避免用户点击后 iOS 弹系统级"无法打开"对话框造成视觉破窗）。建议视觉处理：
  - 链接行带一条很淡的下划线 + 一个细小的"外链"符号（不必是箭头，可以是手绘的小圆圈、报刊式的破折号——任何贴 brief 的）
  - **悬空可访问性**：tap 之后真去尝试打开；失败由 toast/whisper 提示，不要在卡片本身画"❌ 无法打开"那种状态视觉

### b. 用户贴 http 图床链接

如果用户从浏览器复制了一个 `http://` 的图床链接进来当 attachment 引用——同上链路，能升 https 就升，否则 chip 会带一个降级 hint。设计上**不要画"失败 attachment"的专属错位视觉**，统一走"hint 一行小字"就好。

### cd 不能做的事

- **不能假设站点都能打开**。设计里别画"已验证可达"的绿勾。
- **不能在文案上写"点击查看完整网页"**（万一打不开很尴尬）。中性表达，比如"看一眼来源"。

## 3 · WKWebView 文件路径 —— 图片到底是怎么显示的

cd 看到 `<img src="...">` 的时候，`src` 里可能是这三种字符串之一，**对你的视觉判断不影响**，但你要知道它们存在，免得对"图片为什么有时秒出有时延迟"产生错误的设计假设：

| `src` 类型 | 出现场景 | 显示行为 |
|---|---|---|
| `data:image/jpeg;base64,/9j/...` | **当前浏览器 mock 阶段**所有 attachment | 同步秒渲，浏览器直接 decode |
| `capacitor://localhost/...` 或 `file:///var/.../attachments/...`（经 `Capacitor.convertFileSrc` 处理） | **真机阶段**所有本地 attachment | 略慢一拍（从沙盒文件读），但仍然算"瞬时" |
| `https://...` 远端 URL | web_search 召回的外链图（不常见，目前 chat 流里很少） | 取决于网络 |

**对设计的影响**：

- chip / 引用块 / journal 内嵌缩略图都要画**载入态**（同图位置先有一个低饱和占位、然后图浮上来），因为 dataURL 转 capacitor:// 切换的那一刻有真实延迟。占位可以是纸纹理、印章未盖满、空相框——任何贴 brief 的"还没显形"语言。
- **不要为不同 `src` 类型画不同视觉**，dev 看不出来的，用户更看不出来。

### cd 不能做的事

- 不能让某些视觉元素**只在 https 远端图加载完成后才出现**（这是 web 时代的设计习惯，移动端绝大多数图是本地的，没有"网图加载"那种延迟可玩）

## 4 · 触屏交互 —— 没有右键、没有 hover

移动端的交互词汇表跟桌面**不重合**。下面这套是 mobile-api.js / mobile.js 已经定型的惯例，cd 出稿沿用：

### 触发动词表

| 动词 | 时长 / 阈值 | 用途 | 视觉反馈 |
|---|---|---|---|
| **tap** | 瞬时 | 主操作（勾选、打开、选择） | 系统级高亮 / 按下涟漪由 cd 定 |
| **longpress** | **480ms 已锁死** | 唤出"action sheet / 管理面板 / 把这条拉进对话" | longpress arming 中要有渐变视觉（不要 0→1 突变），到时间触发**轻震动**（`navigator.vibrate(12)`） |
| **横滑（水平 swipe）** | 阈值 dev 处理 | 打卡卡片切换条目、日期带切今天/昨天 | 容器 `-webkit-overflow-scrolling: touch` 滚动惯性 |
| **下拉刷新** | 暂不使用 | — | 不要设计下拉刷新（mobile 当前没有刷新语义，数据写就刷） |

**没有 hover**。任何依赖 hover 才能看到的二级信息（桌面端的悬停提示、悬停高亮）在 mobile 都消失。需要的话 → 改成 longpress 唤出。

### 触控目标尺寸

**Apple HIG 硬下限：可点击元素 44×44 pt（约 44×44 px）**。

- 任何看起来很小的 icon（水杯打卡、补剂打卡、tag 行的 ×）的**真实命中区**必须 ≥ 44×44。
- cd 视觉上可以画 24×24 的小图标，但**周围至少包 10px 透明 padding**。在设计稿上把这个 padding 标出来，dev 才不会按视觉边界切实际命中区。
- 横滑容器里的卡片 / chip：**横向至少 60px**，避免相邻两条挤在一起一指头点中两个。

### longpress 的语义

longpress 是 mobile 的**唯一二级菜单入口**。在 brief 的"action sheet"语境里它就是右键的替代品。

cd 设计 action sheet 时：
- 不要做成 iOS 系统原生那种从底部弹起的圆角白卡（太 dashboard 太 SaaS）
- **遵循 brief 的"像纸"调性**——可以是从被长按元素位置展开的一张小纸条、一份夹批、半透明便签——但要确保选项是**触屏友好**的（每行高 ≥ 44px、文字 ≥ 15pt）

### cd 不能做的事

- 不能在视觉里依赖 hover 状态（移动端没这个概念）
- 不能把可点击区域画得**比 44px 小且无外部 padding**（命中失败 = 用户骂街）
- 不能改 longpress 480ms 这个时长（产品惯例已经写进 mobile.js，所有用户已经形成肌肉记忆）

## 5 · 隐私 + 性能预算 —— 别让用户的 iPhone 撑爆

mobile 端的内存、存储、上行带宽**都是真实有上限的**，桌面端的"狂贴图 + 长 entry"在 iPhone 上要被收着设计。

### a. iCloud Backup 风险

- iCloud 免费档 **5GB**。
- 用户的 vault 走 App Documents 目录默认进 iCloud Backup。
- 用户**狂贴 30+ 张照片**两个月就能撑满。撑满之后**整台 iPhone** 的备份都会失败，用户会收到 Apple 推送，把锅算到本 app 头上。

**对 cd 的影响**：
- attachment 区的视觉**不要鼓励海量贴图**。chip 列里多张图的视觉处理，建议保持"克制感"——**1–3 张是常态，4 张以上要让画面有"塞太多了"的视觉负重感**（不是禁止，但要让用户自发感觉"够了"）
- 拖图入口的提示文案 / icon 不要画"批量上传 / 多选"那种 SaaS 习气
- **不要设计相册式 grid**（4×N 网格 = 怂恿用户把整本相册搬进来）

### b. base64 内存爆炸

- WKWebView 的内存上限大约 **1.4GB**（iPhone 12 实测），但当前进程能用的可能只有 ~700MB。
- 一张 5MB 的 JPG 转 base64 dataURL 后约 **6.7MB 字符串**，decode 成像素位图再吃 **20MB+**。
- 一次性渲 5 张这样的图 → **可能 OOM 闪退**。

**对 cd 的影响**：
- chat thread / journal entry 里**嵌入图必须缩略**——视觉上的缩略图（不是 src 上的，是真渲染尺寸），**目标渲染边长 200×200 以内**。
- 设计的 chip 大小、引用框的图位、journal 内联图都应该是**指甲盖大小**（手指盖得住的尺寸），点击/longpress 进**单独的查看态**（全屏查看由 dev 处理）才走原图。
- **如果你的设计里有"图墙"或"拼贴"那种把多图同时大尺寸铺开的画面，提前跟 dev 对一次**，否则真机会闪退。

### c. HEIC 自动转 JPEG

iPhone 默认相机存 HEIC（Apple 新格式），但 web 端解码 HEIC 兼容性差。
**Camera plugin 已锁 `format: 'jpeg'`**，dev 这一侧强转。cd 不用关心 HEIC，所有图都按 JPEG 视觉来出。

### d. 上行带宽

- 4G/5G 下用户拍一张 iPhone 原图 5MB+，上行到 attachment 链路是可观的等待时间。
- 已有 dev 侧的压缩链路，但 cd 设计**上传中态**时要画一个真有时长的过程，不要画"瞬时 ✓"那种欺骗动效。
- 建议视觉：墨水从上往下浸下来 / 印章慢慢盖下 / 纸的纤维一根根浮现——任何**自带等待感**的过渡，让用户感知"它在干活"。

### cd 不能做的事

- 不能鼓励"多图拼贴"或"相册式 grid"这种把存储用量推高的视觉
- 不能让缩略图直接渲染 5MB+ 的 dataURL（设计稿里如果出现缩略图，要明确标注"thumbnail 200×200"让 dev 知道生成缩略图）
- 不能画"上传瞬时完成"的虚假动效

## 6 · cd 绝对不能做的事 —— 一票否决清单

下面这五条，**任何一条出现在交付稿里，dev 都会打回**。不是品味问题，是**会让 app 上架被拒 / 跑不起来 / 闪退**。

1. **不能裸开 NSAllowsArbitraryLoads**
   即"让 app 接受所有 http 明文"。Apple 审核会拒。cd 设计中不要把 http 链接当一等公民——参 §2，全部走 https 升级 + 失败降级 hint。

2. **不能假设相机权限已经给了**
   首次使用 Camera 必须弹系统权限对话框，由用户决定。
   - 文案在 `NSCameraUsageDescription` 里写死，**不能笼统说"用于拍照"**（Apple 会拒），必须具体说明用途，例如"用于把照片贴进当天的日记 entry"。这一行 dev 写，但 cd 如果在视觉里有"拍照"入口，要预设三种态：未询问 / 已授权 / 拒绝后再点。
   - 拒绝后的视觉**不能消失也不能装作没拒绝**——要给一个低强度的"去设置里开权限"路径（一行小字 + tap 去系统设置），不要做成 SaaS 式的红色错误条幅。

3. **不能让 LLM 看到 dataURL 原文**
   一张 5MB 图的 dataURL ≈ 1.5M tokens，**直接爆炸**（context 上限只有几百 K，且 RMB 也烧光）。
   - 当前链路是：dev 把 dataURL 存本地，给 LLM 看的是 **logical URL（`/attachments/<date>/<stem>`）**，LLM 通过 vision tool 用这个 logical URL 反查图片。
   - **cd 的影响**：在设计 chat 流引用图片的视觉时，不要画"AI 正在 OCR 这张图（带原图大缩略）"那种动效——AI 看的不是图本身，是它的"摘要 + reference"。视觉上画"AI 派人去翻这张图"的隐喻就够了（一只手伸进抽屉里、一根线连过去），不要让原图视觉霸屏。

4. **不能违反 DESIGN_BRIEF 的调性铁律**
   重申，因为很多 mobile 端设计稿会"为了让移动端更好用"不自觉滑回 SaaS 习气：
   - **不卡片化**：journal 里每个半小时块不是一张白卡，是纸上的一段；chat 消息不是气泡，是夹批 / 信笺
   - **不 dashboard 感**：widget 不是仪表盘 panel，是"日记本上长出来的"——补剂打卡是夹在内页的小纸片，水杯是页角的钢笔画
   - **像纸**：所有有"白底圆角阴影 + 系统 UI 字"反射的视觉都属于 SaaS 习气，改回纸纹/印记/手绘
   - **AI 是隐形协作者**：tool 调用过程不画 dashboard 式 progress bar，画"墨在纸上化开 / 一行字慢慢浮现"

5. **不能为不同设备出不同视觉**
   不要做"iPhone SE 时切到极简模式 / Pro Max 时多一栏"。
   - 单一视觉，从 320 到 430 都用，靠**横向呼吸 + 内容数量自适应**自然撑开。
   - 拒绝"宽度断点"那种 web 时代的响应式思路——iPhone 屏不够宽到值得多套设计。
